# Architecture

## How Open Arcana works

Open Arcana sits between Claude Code and your Obsidian vault. It configures Claude Code's behavior through three mechanisms:

1. **Rules** (`.claude/rules/*.md`): Loaded into context on every session. They shape how the AI thinks, reasons, and validates its work.
2. **Hooks** (`.claude/hooks/*.sh`): Shell scripts triggered by Claude Code events (SessionStart, PreToolUse, PostToolUse, PreCompact). They enforce constraints and inject context.
3. **Commands** (`.claude/commands/*.md`): Slash commands available during sessions. Each one is a prompt template that the user invokes with `/command-name`.

```
┌─────────────────────────────────────────┐
│  Claude Code Session                     │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │

│  │(context) │  │(events)  │  │(user)  │ │
│  └────┬─────┘  └────┬─────┘  └───┬────┘ │
│       │              │            │      │

│           ┌──────▼──────┐                │

│           │(master cfg) │                │

│           │ Your Vault  │                │
│           └─────────────┘                │
└─────────────────────────────────────────┘
```

## Event flow

### Session start
1. `session-scan.sh` fires (SessionStart hook)
2. Scans for recently modified files, reports changes
3. Loads hot-cache, concept-index stats, and rule counts
4. Claude Code auto-loads all `.claude/rules/*.md` into context
5. CLAUDE.md is loaded as project instructions

### When writing a note
1. User (or AI) triggers Edit/Write on a vault file
2. `prefetch-context.sh` fires (PreToolUse): injects domain-specific context
3. `validate-frontmatter.sh` fires (PreToolUse): checks YAML structure
4. `memory-injection-scan.sh` fires (PreToolUse): blocks injection patterns
5. The write happens
6. `validate-write.sh` fires (PostToolUse): validates content
7. `cascade-check.sh` fires (PostToolUse): reminds about cross-references

### Before context compaction
1. `memory-nudge.sh` fires (PreCompact)
2. Reminds the AI to save important learnings to memory files before context is compressed

## Module dependency graph

```
Core (always installed)
  │
  ├── anti-sycophancy     (standalone, no dependencies)
  ├── token-efficiency     (standalone, no dependencies)
  │
  ├── enforcement-hooks    (needs core hooks infrastructure)
  ├── security-hooks       (needs core hooks infrastructure)
  │
  ├── vault-structure      (standalone, creates folders + templates)
  │     │
  │     └── retrieval-system  (works best with vault-structure)
  │           │
  │           └── commands    (some commands use retrieval)
  │
  ├── connected-sources    (standalone config templates)
  ├── scheduled-tasks      (standalone patterns)
  └── vault-health         (works best with vault-structure)
```

No module has hard dependencies on another. But some work better together. The setup wizard installs them independently; the recommended combinations are in the `examples/` directory.

## File ownership

Open Arcana only creates files in these locations:
- `.claude/rules/` (rule files)
- `.claude/hooks/` (hook scripts)
- `.claude/commands/` (slash commands)
- `.claude/settings.local.json` (hook wiring)
- `.claude/arcana.config.yaml` (module state)
- `CLAUDE.md` (master config, root of vault)
- `00-Dashboard/` (retrieval system dashboards)
- `80-Templates/` (note templates)

Your notes, daily notes, MOCs, and other vault content are never touched by setup or uninstall.

## Configuration

All user-specific values are stored in `.claude/arcana.config.yaml`, generated by the setup wizard. This file is in `.gitignore` by default (it contains your paths and optional API IDs).

Template variables (`{{USER_NAME}}`, `{{VAULT_PATH}}`, etc.) are replaced at install time. To change a value after installation, either re-run `./setup.sh` or edit the generated files directly.
